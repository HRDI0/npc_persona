02 / Handoff Map

인수인계 관점의 읽기 순서

1. 목적GraphRAG MVP는 NPC가 아무 사실이나 말하지 않게 만든다.
2. 원천NPC frontmatter와 story-chunk가 지식 단위가 된다.
3. 그래프NPC, Quest, Clue, Truth, KnowledgeChunk가 관계로 묶인다.
4. 런타임역할, 상태, 힌트 레벨이 조회 범위를 제한한다.
5. 운영Docker Compose와 환경변수로 서버 반영 절차를 고정한다.

처음 보는 사람은 기능 이름보다 데이터 흐름을 먼저 보아야 한다. 이 지도는 `PROJECT_HANDOFF_REPORT.md`의 순서를 시각화한 것이다. 원천 데이터 원칙을 먼저 이해하면, import 경로와 Streamlit 조회 조건이 자연스럽게 연결된다.

03 / Architecture

Streamlit, Neo4j, vLLM은 서로 다른 책임을 가진다

Source: Markdown + YAML
frontmatter parse / story-chunk fence scan
Importer: validation + relationship upsert
MERGE nodes, rebuild owned outgoing edges
Neo4j: NPC-[:KNOWS]->KnowledgeChunk
Cypher gate by role, hint, quest state
Streamlit: prompt assembly + streaming response
vLLM: /v1/chat/completions, stream=true

아키텍처의 핵심은 모델 앞에 지식 선택 단계를 둔 점이다. 전체 DB를 모델에 넣지 않고, `get_allowed_chunks`가 고른 근거만 `build_prompt`에 들어간다. 그래서 NPC 성격보다 먼저 데이터 권한이 답변 범위를 결정한다.

04 / Source Folder

원천 폴더는 사람이 편집하기 쉬운 단위로 나뉜다

rsc/data/npcs/*.md프로필, 말투, 제한 지식, story-chunk
rsc/data/quests/*.yaml퀘스트 상태, required clue, story_expansion
rsc/data/world/*.yamlRole, Event, Clue, Truth 기준 ID
rsc/data/locations/*.mdLocation 노드의 설명과 분위기

스토리 변경은 생성 산출물이 아니라 `rsc/data`에서 시작한다. NPC 파일은 개인 지식의 범위를 담고, quest YAML은 진행 단계와 공개 정책을 담는다. 이 분리는 운영자가 어떤 정보를 어디에 추가해야 하는지 빠르게 판단하게 한다.

20 / Runtime Filter

Cypher gate는 NPC, 역할, 힌트, 상태를 동시에 본다

MATCH (:NPC {npc_id: $npc_id})-[:KNOWS]->(k:KnowledgeChunk) $player_role IN k.allowed_roles k.hint_level <= $allowed_hint_level k.answer_sensitive = false OR $quest_state IN ["ready_to_answer", "solved"]

MATCH (:NPC {npc_id: $npc_id})-[:KNOWS]->(k:KnowledgeChunk), $player_role IN k.allowed_roles, k.hint_level <= $allowed_hint_level, k.answer_sensitive = false OR $quest_state IN. 이 조건이 정답 공개를 코드 수준에서 늦춘다.

21 / Gates

런타임 gate는 세로로 쌓이는 검문소처럼 동작한다

NPC gate선택 NPC가 KNOWS로 연결한 chunk만 후보
Role gateplayer_role이 allowed_roles에 포함되어야 함
Hint gatehint_level이 허용 레벨 이하일 때만 통과
Reveal gateanswer_sensitive는 ready_to_answer 이후에만 통과

각 gate는 독립 장식이 아니라 실제 Cypher WHERE 조건의 시각화다. 하나라도 통과하지 못한 chunk는 prompt assembly 단계에 들어가지 않는다. 이 방식은 NPC 성격과 퀘스트 진행을 같은 조회 과정에서 묶는다.

process-swimlane은 원천 파싱부터 응답 확인까지의 책임을 레인으로 나눈다. 각 레인은 실패 지점도 다르다. 그래서 문서에는 source, import, runtime, LLM, debug를 한 화면에 놓아 추적 경로를 남긴다.

33 / Source To Graph

원천에서 그래프로 가는 과정은 검증 가능한 plan을 만든다

read NPC Markdown
frontmatter parse
story-chunk fence scan
read quest YAML
world ID validation
import plan validation
MERGE node
relationship upsert

source-to-graph-detail은 DB 접속 전에 plan을 만들고 검증한다. 이 단계에서 NPC 4명, Quest 5개, KnowledgeChunk 26개, 참조 ID 무결성을 먼저 확인한다. 그 뒤 Neo4j에 노드와 관계를 병합한다.

34 / Runtime Detail

런타임은 UI 상태를 Cypher 파라미터로 바꾼다

sidebar state

Cypher gate

runtime-process-detail은 Streamlit 화면의 선택값이 내부 query로 들어가는 길을 표시한다. `quest_id`, `quest_state`, `player_role`, `allowed_hint_level`은 모두 prompt 문장이 아니라 DB 조회 조건이다.

35 / Feedback Loop

운영 피드백은 원천 수정으로 되돌아간다

Streamlit answer review
debug expander evidence
source edit in rsc/data
dry-run validation
Neo4j merge load

답변이 어긋나면 prompt만 만지는 것이 아니라 원천 지식과 공개 조건을 확인한다. 이 루프는 운영자가 잘못된 답변을 발견했을 때 어느 파일로 돌아가야 하는지 보여 준다. 생성 산출물은 수정 대상이 아니라 검증 결과다.

36 / Production Apply

서버 반영은 pull, env, build, up, import 순서다

git pull
copy env
build streamlit
up services
source import
verify

운영 DB에는 기본적으로 --reset을 붙이지 않는다. 운영 반영은 기존 데이터를 전체 삭제하는 절차가 아니라 `MERGE` 기반 병합 적재를 기본으로 한다. reset은 분리된 검증 DB처럼 승인된 환경에서만 선택한다.

37 / Commands

반영 명령은 반복 가능한 블록으로 고정한다

git pull --ff-only origin main cp .env.example .env docker compose --env-file .env build streamlit docker compose --env-file .env up -d neo4j streamlit uv run --frozen python src/db_control/import_story_source_to_neo4j.py --source-dir rsc/data

명령은 서버에서 main을 빠르게 당긴 뒤, `.env`를 만들고 Streamlit 이미지를 다시 빌드하는 순서다. 원천 적재 명령에는 --source-dir rsc/data를 명시한다. 운영 DB에는 기본적으로 --reset을 붙이지 않는다.

38 / Environment

환경변수는 DB와 모델 접속면을 고정한다

NEO4J_URIbolt endpoint
NEO4J_PASSWORDsecret in env only
VLLM_URLOpenAI-compatible endpoint
MODEL_NAMEgoogle/gemma-4-E4B-it

NEO4J_URI, NEO4J_PASSWORD, VLLM_URL, MODEL_NAME, google/gemma-4-E4B-it. 값은 코드에 박지 않고 환경변수로 둔다. 모델명은 Streamlit payload와 vLLM 서버가 같은 대상을 보도록 맞춘다.

39 / Verification

분리 검증 스택은 운영 포트와 충돌하지 않는다

docker compose --env-file .env.design-test.example -f compose.design-test.yaml --profile gpu up -d http://127.0.0.1:17474 bolt://127.0.0.1:17687 http://127.0.0.1:18501 http://127.0.0.1:18000/v1/models neo4j / admin2026

검증 스택은 Neo4j Browser, Bolt, Streamlit, vLLM API의 host port를 별도로 쓴다. 이 값은 실제 캡처의 접속면과 일치한다. 운영 서버에서는 기존 컨테이너 주소에 맞춰 환경변수만 바꾸면 된다.
